# Architecture Rules

These rules describe how SiteLog is put together. Every agent building features must follow this architecture. Do not introduce new patterns without discussing them with the developer first.

## The Stack

SiteLog is a Next.js App Router application in TypeScript, backed by PostgreSQL through Prisma. Styling is CSS Modules driven by design tokens. No separate backend — everything lives in the Next.js app via server components, server actions, and route handlers. No payment provider, no webhook receiver. The only outbound integration is SMTP email (Nodemailer) for the password-reset code. Every data page sits behind authentication; the only public pages are the marketing/auth screens.

- Next.js (App Router), React, TypeScript
- PostgreSQL via Prisma
- CSS Modules, with all design values pulled from `tokens/design-tokens.css`
- Object/blob storage for the one image a report or issue can carry (Phase 2 feature — use a free-tier provider; Vercel Blob recommended)
- Vercel for deployment
- Light mode only

## Directory Layout

The full project tree is in `agents.md`. The directories that matter for architecture decisions:

- `tokens/design-tokens.css` — the styling source (all CSS variables); regenerated by `tokens/generate-tokens.js`.
- `prisma/schema.prisma` — the single source of truth for the database; migrations in `prisma/migrations/`.
- `src/app/` — route groups: `(auth)` (one conditional auth page), `(dashboard)`, `(reports)`, `(issues)`, `(materials)`, `(settings)`, plus `api/` for route handlers (auth, attachments) and the root `layout.tsx`.
- `src/components/` — `ui/` primitives, `states/` (EmptyState/LoadingState/ErrorState), `features/` (KpiCard, ReportForm, etc.).
- `src/lib/` — `prisma.ts` (client singleton), `auth.ts` (getSession), `permissions.ts` (server-side checks), `validation.ts` (zod schemas), `constants.ts` (domain constants), `env.ts` (env validation), `storage.ts` (image upload), `email.ts` (Nodemailer — sends the password-reset code), `activity.ts` (the Recent Activity read query).

## Role-Based Rendering

SiteLog has two roles that see different things. The role lives on the session. After login, the layout reads the session role and renders the correct navigation (PM sidebar vs supervisor nav — see agents.md). Pages themselves check the role server-side before rendering role-specific content. The supervisor must never receive PM-only data in a payload, even if the UI would hide it — filter it out on the server. Hiding a nav item is cosmetic; the server decides what data the user actually gets.

## Rendering Rules

Server components are the default. Pages that read and display data — the dashboard, the project detail page, the report/issue/request lists, every detail view — render on the server and fetch their data directly through Prisma. Do not fetch from an API route inside a server component when you can query the database directly.

This is a performance rule, not just a style preference. Marking pages `"use client"` unnecessarily, or fetching data from the browser when it could be fetched on the server, ships more JavaScript and makes pages load slower. Default to server rendering so pages arrive fast with their data already filled in. Reach for the client only where the interaction genuinely requires it.

Client components are only for interactivity. The report form, the issue form, the material request form, the project creation form, the filter controls, and the notification bell dropdown are client components because they hold state and handle events. Mark them with `"use client"` and keep the client boundary as low in the tree as possible. A list page that is mostly static but has a filter bar should keep the page as a server component and make only the filter bar a client component.

The daily report form preserves its input in the browser's localStorage as the supervisor types, and clears it on successful submit. This is a client-side concern handled inside the report form component. There is no draft state on the server and no draft row in the database.

## App Shell and Page Composition

Authenticated pages share one persistent shell. The sidebar (PM) or nav (supervisor) and the top bar live in a shared `layout.tsx`, and individual pages render inside it as `children`. When the user navigates between sidebar items — Projects, Issues, Reports, Material Requests — the shell stays mounted and only the content area swaps. The sidebar must not reload, flicker, or re-render on navigation.

Each sidebar destination still has its own route and URL (`/projects`, `/issues`, `/reports`, `/material-requests`) so every view is bookmarkable, shareable, and survives a refresh. Having routes is correct — what must not happen is a full-page reload or a rebuilt shell on each click. The shared layout is the mechanism: persistent shell, swapping content, real URLs underneath.

## Auth Page

There is one auth page, not separate pages per form. The login, sign-up, and password-reset forms all live on a single auth screen, rendered conditionally. Switching forms swaps in place — no navigation, no full reload. This keeps auth consistent and avoids duplicating the auth layout.



## Data Flow

Every page in SiteLog is authenticated, so there are no public write paths. There are two kinds of writes:

1. **Mutations from the authenticated UI** go through server actions. This covers everything a logged-in user does through a form or button: submitting a report, creating an issue, updating an issue status, creating a project, assigning a supervisor, submitting a material request, approving or rejecting a request, marking a request fulfilled, editing a profile. The form calls a server action, the action validates input with zod, checks permissions, writes through Prisma, fires any notification, and revalidates the affected cache.

2. **Auth and file upload go through route handlers** under `app/api/`. Login, signup, logout, and password reset (`forgot-password`, `reset-password`) are route handlers in `app/api/auth/`. Image upload is a route handler in `app/api/attachments/` (multipart file data). Use a route handler only when a server action will not do — auth boundaries, file uploads, or anything called by client-side fetch.

Reads do not go through either. Server components query Prisma directly.

**Recent Activity is a read query, not a write path and not a table.** The activity feed on the dashboard and on the project detail page is built in `lib/activity.ts` by querying the most recent DailyReports, Issues (created and resolved), and approved MaterialRequests, merging them, sorting by date, and returning the top N. There is no ProjectActivity table. Nothing writes activity rows. When a report is submitted or an issue is resolved, no extra activity record is created — the activity feed simply reads from the existing tables.

## Notifications

When a mutation should notify the PM (report submitted, issue created, material request submitted), the same server action creates the Notification row inline — no queue, no background job, no event bus. The bell reads the recipient's notifications; the count clears when the dropdown opens. Three triggers only.

## State Management

There is no global state library. React state, server data, and URL search params are enough. Filters on list pages are held in the URL as search params so a filtered view is shareable and survives a refresh, and so the server component can read the filter and query accordingly. If you feel the urge to add Redux, Zustand, or Jotai, stop and reconsider — the app is simple enough that `useState` plus server actions plus search params cover every case.

## Database Access

All database access goes through Prisma. Raw SQL is only allowed inside migration files. Every query that takes user input uses Prisma's query builder, never string interpolation.

The Prisma client is imported from `lib/prisma.ts`, which exports a singleton. Do not call `new PrismaClient()` anywhere else; multiple clients exhaust the connection pool in development.

Data is always scoped by access. A supervisor's queries are filtered to the projects they belong to (via ProjectMembership) and, for reports and requests, to their own records. Never return rows the requesting user is not permitted to see, even if the UI would hide them.

## Authentication

Sessions are cookie-based. Cookies are `httpOnly`, `secure` in production, and `sameSite: lax`. The session helper in `lib/auth.ts` exposes `getSession()` for server components and server actions. The session carries the user id and role. Client components that need auth state receive it as a prop from a parent server component — they do not read the session directly.

Passwords are hashed before storage. Sessions expire and re-authenticate gracefully without the user losing form data (the report form's localStorage draft protects against this). Full rules are in `security.md`.

**Email and password reset.** SiteLog sends one kind of email: the password-reset code. Sending goes through `lib/email.ts`, which wraps Nodemailer over SMTP (credentials in the `SMTP_*` env vars — Phase 1 can use a free SMTP account such as a Gmail app-password; no domain required). The forgot-password flow is code-based, not link-based: a 6-digit code is generated, stored on the User row with a 15-minute expiry, emailed, and verified inside the app (see security.md). The reset route handlers live in `app/api/auth/` (`forgot-password` to request a code, `reset-password` to verify and set the new password).

## Image Uploads

Image upload is a Phase 2 feature — do not build it until the Phase 1 core product works end to end. The full product can be built and demoed without it; reports and issues keep all their text fields with or without an image.

When built: a report can carry one image, an issue can carry one image, material requests carry none. Use a free-tier blob provider (Vercel Blob recommended — its free Hobby tier covers a capstone with no billing risk). Images upload through the `app/api/attachments/` route handler, validated server-side by actual file bytes (not the client's MIME claim), capped at 5MB, stored via `lib/storage.ts`, and referenced by URL in the Attachment table. Never store images on the filesystem (ephemeral on Vercel). Never trust the client filename or content type.

## Error Handling

Server actions and route handlers return structured results. Success is `{ ok: true, data }`. Failure is `{ ok: false, error: { code, message } }`. The client never receives a raw exception message, stack trace, or Prisma error object. Log the full error on the server; return a sanitized, user-friendly message that matches the copy in `design-system.md`.

## Environments

- **development** runs locally against a local PostgreSQL database.
- **production** runs on Vercel against the production database.

All secrets (database URL, session secret, SMTP credentials, storage token) are environment variables, never committed. No staging environment for the capstone.

## What Not to Do

- Do not add Tailwind, styled-components, or any CSS-in-JS. CSS Modules with `tokens/design-tokens.css` only.
- Do not add GraphQL. Server actions and a few route handlers are enough.
- Do not add a separate Node or Express backend. Everything stays in the Next.js app.
- Do not reach for microservices, a message queue, or an event bus. Notifications are written inline.
- Do not create a ProjectActivity table. Recent Activity is a read query.
- Do not add a global state library.
- Do not fetch from your own API routes inside a server component — query Prisma directly.
- Do not store uploaded images on the filesystem. Use the storage provider via `lib/storage.ts`.
- Do not instantiate `new PrismaClient()` outside `lib/prisma.ts`.
- Do not build dark mode, a payment flow, webhooks, or public data pages. None exist in SiteLog.
